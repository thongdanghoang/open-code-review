OpenCodeReview như lớp review độc lập cho code Codex sinh ra

Bạn để Codex viết code mỗi ngày và không còn đọc từng dòng. Câu hỏi còn lại: ai review code đó? ocr chèn vào đúng khe hở này — một bộ review tất định, đặt đúng vị trí từng dòng, độc lập với agent đã viết ra code.

Số liệu benchmark của dự án trên 200 PR thực, 50 repo, 10 ngôn ngữ. Điểm cần nhớ: công cụ này được tinh chỉnh cho độ chính xác, không phải để thay thế toàn bộ review.

Vì sao hợp với bạn

Khe hở trong quy trình "không tự code"

Khi Codex sinh code, hai lựa chọn review đều có điểm yếu: tự bạn đọc từng dòng thì mất đi cái lợi của việc dùng AI; còn để chính Codex tự review thì nó thiên vị với chính sản phẩm của mình và hay cắt góc trên changeset lớn.

ocr giải quyết bằng kiến trúc tất định × agent: những bước không được sai do code thường xử lý, phần suy luận mới giao cho LLM.

  • Chọn file tất định — quyết định chính xác file nào review, file nào lọc; không bỏ sót thay đổi quan trọng vì agent "lười".
  • Gom file liên quan thành một đơn vị review chạy isolated, ổn định trên changeset rất lớn và review song song được.
  • Module định vị riêng — comment bám đúng dòng, không bị "trôi" số dòng như khi bảo agent review tự do.
  • Match rule theo loại file bằng template engine — tập trung sự chú ý của model, cắt nhiễu từ gốc.

Mô hình tư duy

Đừng coi ocr là "một agent review nữa". Coi nó là cái cổng kiểm tra (gate) chạy sau khi Codex xong việc — giống linter có não. Giá trị lớn nhất đến khi nó chạy tự động, không phụ thuộc bạn nhớ gọi.

Cài đặt · làm một lần

Đưa ocr chạy được trong 3 bước

Cài CLI

Đã làm xong trên máy bạn. Đây là lệnh để cài trên máy khác / CI.

bash
# cài global, có lệnh `ocr` ở mọi nơi
npm install -g @alibaba-group/open-code-review
ocr version   # → open-code-review v1.6.5

Cấu hình LLM (bắt buộc)

ocr dùng API key LLM riêng của nó — tách khỏi backend của Codex. Chọn một trong các kiểu dưới. Với công ty, kiểu C (gateway nội bộ) thường là cái bạn cần.

A · Anthropic / OpenAI trực tiếp

bash · non-interactive
ocr config set provider anthropic
ocr config set providers.anthropic.api_key sk-xxx
ocr config set providers.anthropic.model claude-sonnet-4-6

B · Provider tương thích OpenAI (DeepSeek, Kimi, Qwen/DashScope, z-ai…)

bash
ocr config set provider deepseek
ocr config set providers.deepseek.api_key sk-xxx
# xem toàn bộ provider dựng sẵn:
ocr llm providers

C · Gateway nội bộ công ty (private endpoint tương thích Anthropic/OpenAI)

bash · custom provider
ocr config set provider company-gw
ocr config set custom_providers.company-gw.url https://llm.intra.elca/v1
ocr config set custom_providers.company-gw.protocol openai
ocr config set custom_providers.company-gw.api_key your-key
ocr config set custom_providers.company-gw.model gpt-4o

Config lưu ở ~/.opencodereview/config.json. Trong CI, dùng biến môi trường (OCR_LLM_URL, OCR_LLM_TOKEN, OCR_LLM_MODEL, OCR_USE_ANTHROPIC) — ưu tiên cao nhất, không cần ghi file.

Test kết nối & chạy thử

bash
ocr llm test                # kiểm tra LLM thông chưa
ocr review --preview      # xem file nào sẽ được review — KHÔNG tốn token
ocr review                  # review thật toàn bộ thay đổi workspace

Mẹo

Luôn dùng --preview trước trên repo lạ hoặc changeset lớn để biết phạm vi (và ước lượng chi phí) trước khi gọi LLM thật.

Hiểu lầm phổ biến · đọc trước khi cấu hình

Gói Claude Team/Pro không chạy được OCR

Đây là chỗ gần như ai triển khai cho team cũng vướng. Tóm lại: OCR cần một API key thật sự, không "ăn ké" được subscription Claude.

Anthropic có hai sản phẩm tách biệt, dùng hệ xác thực và hệ tính tiền khác nhau:

Xác thựcĐăng nhập OAuth (theo ghế)API key sk-ant-…

ocr là một CLI Go độc lập — nó KHÔNG phải Claude Code. Nó nói chuyện thẳng với Messages API và cần một API key của Anthropic Console (hoặc provider tương thích OpenAI). Subscription Team/Pro chỉ "nuôi" được Claude Code và claude.ai — không cấp quyền truy cập API cho tool ngoài. Bao nhiêu ghế Claude Team cũng không biến thành "credit API" cho OCR.

Đừng dùng "hack" token Claude Code

README có ghi OCR tương thích biến ANTHROPIC_AUTH_TOKEN của Claude Code, nên về lý thuyết có thể trích xuất OAuth token (chạy bằng subscription) rồi nhét cho OCR. Không nên: token subscription chỉ dành cho client chính chủ — dùng cho tool bên thứ ba là vi phạm điều khoản, lại ngắn hạn nên hay hết hạn, workflow dễ gãy. Không bền cho công ty.

Vậy dùng nguồn LLM nào?

Một trong ba nguồn "đúng là API" — đều cấu hình ở Bước 2 phía trên:

  • Anthropic Console API key — đăng ký riêng tại Developer Platform, nạp credit pay-as-you-go. Đường chính chủ nếu muốn dùng model Claude (claude-sonnet-4-6, claude-opus-4-8…).
  • Provider rẻ hơn tương thích OpenAI — DeepSeek, Qwen/DashScope, Kimi… (xem ocr llm providers). Chi phí biên thấp.
  • Gateway nội bộ công ty — nếu ELCA đã có proxy LLM riêng. Thường là lựa chọn đúng nhất về bảo mật + chi phí (phần custom_providers).

Tin tốt về chi phí

Theo benchmark, OCR tốn ~1/9 token so với để agent thường review — nên một API key pay-as-you-go thường rất rẻ cho việc review hằng ngày.

Chốt chặn ở CI/CD

Đòn bẩy chất lượng thật sự

Đây mới là nơi nâng chất lượng đầu ra một cách hệ thống: mọi PR/MR đều bị review tự động, không phụ thuộc việc bạn có nhớ gọi hay không. Repo có sẵn ví dụ cho GitHub Actions và GitLab CI.

bash · lệnh lõi cho CI
ocr review \
  --from "origin/main" \
  --to "$COMMIT_SHA" \
  --format json        # máy đọc được, parse trong script CI

Dùng commit SHA cho --to để xử lý đúng PR từ fork. Tham khảo examples/github_actions/ và examples/gitlab_ci/ trong repo.

Rule riêng cho team

Đòn bẩy dài hạn

Codex không biết "luật ngầm" của codebase công ty. Rule file biến những luật đó thành kiểm tra tự động. Đặt .opencodereview/rule.json ở gốc repo và commit vào git để cả team dùng chung.

.opencodereview/rule.json
{
  "rules": [
    {
      "path": "**/*.java",
      "rule": "Mọi method mới phải validate tham số bắt buộc cho null",
      "merge_system_rule": true
    },
    {
      "path": "**/*mapper*.xml",
      "rule": "docs/sql-rules.md"   // trỏ tới file .md
    }
  ],
  "exclude": ["**/generated/**", "vendor/**"]
}
  • merge_system_rule: true → ghép rule dựng sẵn cộng rule của bạn. Mặc định (không có) rule của bạn thay thế rule hệ thống.
  • Trường rule nhận cả nội dung inline lẫn đường dẫn file .md — tiện cho rule dài, version-control riêng.
  • Kiểm tra rule nào áp cho 1 file: ocr rules check <path> hoặc ocr rules check --rule custom.json <path>.

Tôi đã kiểm chứng trên repo này

Những gì chạy đúng như tài liệu

Tôi tạo vài thay đổi giả trong chính repo open-code-review, chạy các lệnh không tốn token, rồi dọn sạch. Kết quả:

Chọn file — ocr review --preview

✓version.go (file Go thật) → được review
✕README.md → loại, lý do unsupported_ext (markdown không review mặc định)
✕*_test.go → loại, lý do default_path (file test bị lọc sẵn)

Rule theo loại file — ocr rules check

✓.ts / .tsx có rule riêng: React hooks, async/await, XSS, cấm any / var / ==
✓*mapper*.xml có rule SQL injection (phân biệt ${} nguy hiểm vs #{} an toàn)
✓Custom rule với merge_system_rule:true hiện đúng cả "System-Specific Rules" lẫn "User-Specific Rules"

Quét toàn file — ocr scan --preview

Trên cmd/opencodereview/: chọn đúng 14 file nguồn, loại 15 file _test.go, và in ước lượng token trước khi chạy — nên kiểm soát được chi phí trên repo lớn (kết hợp --max-tokens-budget).

Cần biết trước khi tin tưởng

Giới hạn — nói thẳng

Recall thấp là chủ ý

Công cụ này bắt lỗi chắc-chắn-đúng, không bắt hết. Đừng coi nó là lưới an toàn duy nhất. Việc bạn hoàn toàn không đọc code nữa vẫn là rủi ro — ocr giảm rủi ro đó chứ không xóa bỏ. Logic nghiệp vụ sai mà cú pháp đúng thì bộ review nào cũng dễ bỏ qua.

Cần key LLM riêng & tốn chi phí

ocr không "ăn ké" backend của Codex — bạn trả tiền token riêng cho nó. Tin tốt: theo benchmark nó tốn ~1/9 token so với để agent thường review, nên chi phí biên thấp. Vẫn nên đặt --max-tokens-budget khi scan repo lớn.

Markdown / file không hỗ trợ bị bỏ qua

Mặc định chỉ review các đuôi file lập trình được hỗ trợ; tài liệu, file test bị lọc. Cần review chúng thì cấu hình include trong rule file để bypass bộ lọc mặc định.

Đề xuất triển khai

Lộ trình áp dụng cho team của bạn

  • Tuần 1 — cá nhân. Cài plugin Codex, dùng @Open Code Review sau mỗi thay đổi Codex sinh ra. Cảm nhận chất lượng comment trên codebase thật của bạn.
  • Tuần 2 — rule team. Viết .opencodereview/rule.json đầu tiên với 3–5 luật ngầm quan trọng nhất của dự án, commit vào repo. Dùng ocr rules check để verify rule match đúng file.
  • Tuần 3 — cổng CI. Thêm ocr review --from origin/main --to <sha> --format json vào pipeline ở chế độ cảnh báo (không block merge). Quan sát tỉ lệ báo đúng/sai trong 1–2 tuần.
  • Sau đó — siết dần. Khi đã tin precision, mới cân nhắc để CI fail trên lỗi High. Tinh chỉnh rule liên tục dựa trên false positive thực tế.
